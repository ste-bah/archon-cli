# v1.2.2 - Provider-Aware Auto-Compaction

v1.2.2 remediates the auto-compaction gap tracked in issue #34. The
`config.context.compact_threshold` field is no longer dead — the agent
loop, the subagent runner, and the pipeline LLM adapter all consult a
shared `evaluate_compaction` decision function after each turn, and a
new `LlmError::ContextWindowExceeded` variant lets every wired site
react to provider context-window overflow with a forced compact and
one retry before returning an error.

The TUI status bar now shows context-window usage so operators can see
how full the active loop's context is before it hits the threshold.

## Added

- `LlmError::ContextWindowExceeded { provider_message, provider, model }`
  variant on the now `#[non_exhaustive]` `LlmError` enum. Provider
  mappers translate Anthropic, OpenAI, local OpenAI-compatible, and
  Codex (both RPC and app-server) overflow errors into this variant via
  a shared `archon_llm::context_window::classify_context_window_error`
  helper.
- `archon_llm::context_window::resolve_context_window` four-tier
  resolver: explicit `[context].max_tokens` > provider model metadata
  > known-model heuristic > 200_000 fallback. Killed the hardcoded
  200_000 in `agent/compaction.rs`.
- `archon_llm::usage::UsageAccumulator` consolidates `MessageStart` and
  `MessageDelta` usage events with a `saw_start_usage` guard so cached
  prefix tokens (`cache_creation_input_tokens` and
  `cache_read_input_tokens`) count toward context-window pressure
  exactly once, never double-counted.
- `archon_core::agent::AutoCompactState` circuit breaker (count,
  consecutive failures, disabled, `compact_in_flight` recursion guard)
  with `on_success`, `on_real_failure`, and `on_cancel` lifecycle
  methods. `on_cancel` does not trip the breaker (Ctrl+C during a
  compaction summary is operator-initiated, not a compaction failure).
- `archon_core::agent::evaluate_compaction` shared free function that
  returns `Some(CompactAction::Micro)` at 65% fill, `Some(Full)` at
  the configured threshold, and `None` below. Called from all three
  wired loop sites: main `Agent::process_message`, subagent
  `runner/runtime.rs`, and pipeline `llm_adapter.rs`. Single source of
  truth, no per-site duplication.
- Reactive mid-turn compaction at every wired site: when the LLM
  returns `LlmError::ContextWindowExceeded`, the loop forces a
  compaction and retries the request once. A second failure returns
  the error to the caller instead of looping.
- TUI status bar now shows `context_tokens_used / context_window`
  with percentage on the bottom strip. Aggregated across the active
  loop.
- Match-site audit document
  `docs/development/llm-error-match-site-audit.md` enumerating every
  downstream `LlmError` match site and confirming wildcard-arm safety
  for the new variant under `#[non_exhaustive]`.

## Fixed

- Closes #34. Auto-compaction was broken (config-only, never wired into
  the agent loops). All three loop sites now compact proactively and
  reactively. Long subagent runs no longer kill the session by blowing
  the provider context window.

## Notes

- Auto-compaction currently uses a deterministic synthetic summary plus
  the existing `archon-context` compaction machinery. This is reliable
  and avoids recursive provider failure but trades summary quality for
  safety. A bounded non-agentic LLM summarizer is a follow-up
  improvement.
- Pipeline compaction is per-`ProviderLlmAdapter` (session-scoped
  state via `std::sync::Mutex<AutoCompactState>`). The Mutex is never
  held across `.await` boundaries (mirrors the v1.2.1 Bug-fix-2
  pattern in `AgentHandle::fire_cancel`).
- The TUI context indicator is a basic aggregate display. A
  per-source breakdown (main vs subagent vs pipeline-adapter context)
  is a follow-up if operators need that resolution.
