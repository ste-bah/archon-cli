# v0.1.37 — Tokio Shutdown Panic Fix

Archon CLI panicked on clean exit with `A Tokio 1.x context was found, but it is being shutdown`
when background tasks held in-flight `tokio::time::sleep` futures during runtime teardown.
This release wires cancel-aware `select!` on every async sleep site and introduces lifecycle
guards that own the spawned task handles.

## What's fixed

- **Cron scheduler** — `CronShutdown` lifecycle guard replaces the dead `_cron_cancel` handling.
  `run_scheduler_loop_with_notify` uses `biased; select!` so shutdown notification is checked
  before any sleep. Graceful `.shutdown().await` (2s timeout) with `Drop` abort fallback.
- **SSE reconnect pump** — `SseShutdown` guard + `SseGuardedStream<S>` RAII wrapper own the
  pump's `JoinHandle`. All 3 `tokio::time::sleep` sites now use `biased; select!` against
  a `Notify` that fires on shutdown. `Drop` on the RAII wrapper first drops the channel
  (→ pump observes `ReceiverDropped` and returns), then signals `Notify` + aborts as fallback.
- **SSE handshake timeout** — `ENDPOINT_HANDSHAKE_TIMEOUT` sleep wrapped in `select!` against
  the pump's shutdown `Notify`, preventing the handshake from blocking teardown.

## How shutdown flows now

1. User types `/q` → `run_interactive_session` begins cleanup
2. `cron_shutdown.shutdown().await` — wakes cron, 2s join window
3. MCP transport `mcp_shutdown.shutdown().await` — wakes SSE pump, drops channel,
   pump returns via `ReceiverDropped`, 2s join window
4. Runtime drops → any remaining spawned tasks get `notify_waiters() + handle.abort()`
5. Clean exit, no panic
