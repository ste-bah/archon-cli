# v0.1.43 — fix broken embedding model + add `archon memory reindex`

## What was broken

The local fastembed embedding backend was completely non-functional: `archon docs model-status` returned a smoke-embed failure with:

```
ONNX runtime: Missing Input 'encoder.layer.0.attention.output.LayerNorm.weight'
```

Root cause: archon was hardcoded to use `fastembed::EmbeddingModel::BGEBaseENV15Q` (the **quantized** BGE-base-en-v1.5 variant from `Qdrant/bge-base-en-v1.5-onnx-Q`). The model file shipped from that HuggingFace repo no longer matches what `fastembed = 4.9.1`'s ONNX runtime expects — the runtime requests a LayerNorm weight that's not present in the optimized/quantized graph. Wiping and re-downloading the cache produced the same broken state (the bug is upstream — model file vs. runtime call shape).

Effect: every embed call failed. `archon docs index` could not embed chunks. `MemoryGraph::embed_and_store` silently failed for every memory write. Semantic search across docs and memory was non-functional.

## Fix

Swap to the non-quantized BGE-base-en-v1.5 (`fastembed::EmbeddingModel::BGEBaseENV15`).

| | Before | After |
|---|---|---|
| Model variant | `BGEBaseENV15Q` (quantized) | `BGEBaseENV15` (non-quantized) |
| HuggingFace source | `Qdrant/bge-base-en-v1.5-onnx-Q` | `Qdrant/bge-base-en-v1.5-onnx` |
| Cache size | ~208 MB | ~440 MB |
| Vector dimension | 768 | 768 (no DB schema change) |
| Smoke embed | FAIL | PASS |

Touched two hardcode sites that selected the model:
- `crates/archon-docs/src/embed.rs:94` (document chunk embedder)
- `crates/archon-memory/src/embedding/local.rs:49` (memory graph embedder)

Both log strings updated to drop the now-inaccurate "(quantized)" suffix.

## New: `archon memory reindex --all`

Existing memory embeddings written under the broken (or differently-shaped) prior model are in the wrong vector space. Until v0.1.43 there was no bulk-reindex path — `vector_search::store_embedding` only wrote one memory at a time, called from per-memory write paths. New surfaces:

- **CLI subcommand**: `archon memory reindex --all` — `--all` is required (guard against accidental expensive re-runs)
- **Underlying API**: `MemoryGraph::reindex_all_embeddings() -> Result<(reindexed, skipped, failed), MemoryError>`

Reads every memory in the graph via the existing private `read_all_memories(&db)` helper, runs each through `provider.embed(...)`, upserts the resulting vector via `vector_search::store_embedding(...)`. Mirrors the per-write `embed_and_store` policy: memories shorter than `MIN_EMBED_CHARS` are skipped (they wouldn't have been embedded on write either).

Local verification on the dev machine's persistent memory.db (109 memories):

```
$ archon memory reindex --all
Reindexing 109 memories under provider 'auto'...
Reindexed: 108
Skipped:   1 (content shorter than min-embed threshold)
Elapsed:   44.5s
```

Companion command for docs (already existed): `archon docs index --all` — re-indexes every chunk in the doc store.

## Migration

After upgrading to v0.1.43:

```bash
# 1. Nuke the old quantized model cache so the new variant downloads
rm -rf ~/.local/share/archon/fastembed

# 2. Re-embed all docs under the new model
archon docs index --all

# 3. Re-embed all memories under the new model
archon memory reindex --all
```

First call will take ~30s on a typical machine (model download + cold ONNX runtime init). Subsequent calls are milliseconds.

If you don't run `memory reindex`, semantic memory search (`/memory search ...`, hybrid retrieval in agent context) returns stale results from the old vectors. Keyword recall (`/recall ...`) is unaffected and continues to work.
