# v1.2.1 - TUI Cancellation And TaskCreate Lifecycle Fixes

v1.2.1 is a patch release on top of v1.2.0 that fixes four bugs in the
interactive TUI cancellation chain and the `TaskCreate` tool's task-status
lifecycle. The combination of bugs made the TUI feel hung after a Ctrl+C or
double-Esc: signals were either dropped on the floor entirely (Esc), silently
lost to lock contention (Ctrl+C under load), unable to interrupt running
shell commands (Bash), or unable to surface task progress in `/tasks` after
a `TaskCreate` (background tasks).

## Fixed

- **Double-Esc cancel is now a real cancel.** Previously the double-Esc
  handler in `crates/archon-tui/src/input/dispatch.rs` flipped two display
  flags and printed `"[interrupted]"` but returned `KeyResult::Nothing`. The
  whole cancel chain (`__cancel__` -> `AgentHandle::fire_cancel` ->
  `JoinHandle::abort` -> `CancellationToken` cascade) is wired correctly for
  Ctrl+C but was completely bypassed for double-Esc. Double-Esc now returns
  `KeyResult::SendCancel` when `is_generating`, exactly mirroring the Ctrl+C
  path. Two regression tests added covering "double-Esc during generation
  must return SendCancel" and "double-Esc when not generating must return
  Nothing".
- **`AgentHandle::fire_cancel` no longer silently no-ops on lock
  contention.** The previous implementation used a `tokio::sync::Mutex` with
  `try_lock()`, so a contended slot meant cancellation silently disappeared
  with no log. The slot is now backed by `std::sync::Mutex` because both
  critical sections in `run_turn` only hold the lock for a single
  synchronous mutation (never across an `.await`). Lock acquisition is now
  unconditional, with `tracing::info!` on successful cancel and
  `tracing::debug!` when there is no in-flight turn. Poisoned mutexes are
  recovered via `into_inner` rather than crashing the cancel path.
- **Bash tool now honours `ctx.cancel_parent`.** Previously the Bash tool
  in `crates/archon-tools/src/bash.rs` only enforced its own per-command
  timeout. A subagent stuck in a slow Bash command could not be interrupted
  by Ctrl+C / double-Esc / `/cancel` because the parent
  `CancellationToken` was ignored. The output-reading future is now raced
  against `cancel_parent.cancelled()` via `tokio::select!` with `biased`
  ordering, and on cancellation the child process group is killed and
  reaped so it does not become a zombie. The error path returns
  `"Command cancelled by user"`.
- **`TaskCreate` task lifecycle transitions are now wired.** Previously
  `TaskCreate` called `TASK_MANAGER.create_task` (Pending) but no subsequent
  `set_status` calls were made on either the `ExplicitBackground` or
  `Foreground` paths. Tasks remained `Pending` forever, even after the
  underlying subagent completed. The dispatch sites now transition
  `Pending -> Running` before invoking `run_subagent`, and map
  `SubagentOutcome::Completed/Failed/Cancelled` to
  `TaskStatus::Completed/Failed/Stopped` on return. `AutoBackgrounded`
  leaves the task in `Running` because the inner runner continues in its
  own task and a downstream `on_inner_complete` hook is responsible for
  the final transition.

## Notes

- All four fixes are surgical and do not change any public type signature.
  `AgentHandle::fire_cancel` keeps its existing `fn fire_cancel(&self)`
  shape so both the synchronous `CancelHandler` slash path and the async
  session loop path keep working without modification.
- No new configuration knobs are added.
