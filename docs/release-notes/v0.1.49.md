# v0.1.49 — TUI Drain-Stall False Positive Fix

v0.1.49 stops long document ingests from flooding session logs with false TUI
drain-stall warnings.

The stall detector now checks whether TUI events are actually pending before it
warns. Idle channels are treated as idle, not stuck. Long-running slash
workloads such as `/docs ingest <path>` also get a wider drain threshold while
their mirrored CLI task is running.

Repeated warnings are rate-limited to one per minute per active stall episode,
or sooner if the pending queue grows. Real stalls still warn when queued events
pile up and the render loop does not drain them.

## Other user-visible changes bundled in this release

### Non-TTY interactive fail-fast

`archon` (no subcommand) now refuses to start when stdin or stdout is not a
terminal. Previously the TUI would spin silently on piped input or in CI
runners. The error tells you which non-interactive entry point to use:

```
interactive mode requires a TTY; use -p/--print, --headless, or --ide-stdio
for non-interactive input
```

If you launch `archon` from CI or any non-interactive context, switch to
`archon -p '<prompt>'`, `archon --headless`, or `archon --ide-stdio`.

### Higher default `max_tokens` for OpenAI-compatible VLM

`scripts/archon-init.sh` now seeds `[policy.docs.vlm.openai_compat]` with
`max_tokens = 8192` instead of `1024`. The 1024 default worked for
non-thinking models like Gemma 4 but hard-failed on thinking models
(Qwen3.6, GLM-4.5, etc.) that emit a separate `reasoning_content` field
yet still count those tokens against the cap. With 1024, those models
burned the entire budget on reasoning before producing answer content,
and archon surfaced:

```
VLM provider error (openai-compat): chat completions response did not
contain text
```

8192 leaves comfortable headroom for both thinking and non-thinking
responses on common vision models. Existing projects keep their
configured value; only newly created projects get the new default. For
very verbose thinking models (Qwen3.6 q3_K_XL, etc.) bump higher in
your `.archon/policy.toml`:

```toml
[policy.docs.vlm.openai_compat]
max_tokens = 32000
```

The Rust serde-fallback default in `archon-policy::OpenAiCompatVlmPolicy`
still emits 1024 when an existing policy.toml omits the key — that's a
separate v0.1.50 change since the code-side bump warrants CI
verification.
