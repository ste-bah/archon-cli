# Archon v1.3.12 — Parallel Implementation Fanout

v1.3.12 adds the Workflow Write Coordinator (PRD-012): implementation fan-out
stages can now run multiple items in parallel, safely. Each item executes in an
isolated `git worktree`, non-conflicting items share a wave, and validated
patches are applied back to the canonical repository serially under a
cross-process write lock. Coordination is safe by default and falls back to the
previous serial behaviour whenever it cannot guarantee isolation.

## Added

- `[workflow.write_coordinator]` config block governing the coordinator. New
  keys: `enabled`, `retain_success_worktrees`, `retain_failed_worktrees`,
  `max_patch_bytes`, `max_file_bytes`, `fail_on_undeclared_write`,
  `allow_dirty_canonical_repo` — see
  [config reference](../reference/config.md#workflowwrite_coordinator-prd-012--parallel-implementation-fanout).
- Per-item isolated worktrees, conflict-graph wave scheduling, patch capture +
  validation (declared-only paths, symlink-escape, size budgets, secret scan,
  output usability), and canonical patch apply behind a repository write lock.
- Ten `WriteCoordination*` workflow events plus metadata-only learning rows
  (paths, sizes, and BLAKE3 hashes only — never patch bytes or diff lines).
- A compact `workflow` status block for coordinated stages and resume
  idempotency keyed off persisted per-item apply status.
- Operations guide:
  [Workflow Write Coordinator](../operations/workflow-write-coordinator.md).

## Changed

- Implementation fan-out width is no longer pinned to 1 when the coordinator is
  active; width becomes the floor of the run, policy, stage, runner, and
  subagent caps.

## Migration

- Existing workflows continue working unchanged. The coordinator only engages
  for implementation fan-out stages whose `target_repository_root` is a Git
  repository and whose runner supports a workspace boundary; otherwise the run
  falls back to serial (`non_git_root`, `boundary_unavailable`, or
  `feature_disabled`).
- With `fail_on_undeclared_write = true` (default), implementation fan-out
  stages whose inline items lack both per-item `target_files` and
  `expected_target_files` are now **rejected before launch**. Set
  `fail_on_undeclared_write = false` to retain the prior behaviour; those items
  are then scheduled serially.

## Known limitations

- No auto-merge of semantic conflicts within the same file (NG-WC-002).
- No provider-specific isolation beyond the `StageRunRequest.input`
  `target_repository_root` rewrite (NG-WC-004).
- Applying a patch back to a dirty **tracked** declared target can fail
  `git apply --3way`; commit or stash the conflicting change before the run.
