# v0.1.29 — fix UTF-8 byte-slicing panic in API request body logging

## Bug

`crates/archon-llm/src/anthropic.rs:106` and `:133` used Rust byte-slicing
for log truncation:

    &body[..body.len().min(2000)]

This panics when byte 2000 (or 500) lands inside a multi-byte UTF-8
character. The bug was latent until ARCHON.md was added at repo root in
v0.1.28 — its em-dash content tipped the request body past the threshold,
causing deterministic panics on every API call.

## Fix

Ported Claude Code's `debugBody()` utility from
`project-zero/bridge/debugUtils.ts:46-53` to Rust as a new module:
`crates/archon-llm/src/debug_body.rs`.

The Rust port matches Claude Code's behaviour:
- `DEBUG_MSG_LIMIT = 2000` (same constant)
- Newline collapsing (`\n` → `\\n`)
- Secret redaction (`access_token`, `session_ingress_token`,
  `environment_secret`, `secret`, `token`) with 16-char minimum
- Length suffix: `... (N chars)`

Plus Rust-specific safety:
- Char-boundary-safe truncation (walk back from `min(len, LIMIT)`
  until `is_char_boundary` returns true)
- `once_cell::Lazy` for the regex compilation (no per-call cost)

## Behavioural change

Error-response log cap raised from 500 to 2000 bytes (line 133).
Deliberate — more diagnostic info on API errors. Logging-only change;
request behaviour unchanged.

## Tests

15 new unit tests in `debug_body.rs`:
- 9 truncation cases (short, at-limit, beyond-limit, multi-byte
  boundaries for 2/3/4-byte chars, newlines, empty, single-byte)
- 4 redaction cases (short token, long token partial, all field names,
  non-secret field unchanged)
- 2 integration cases (combined redaction+truncation, regression for the
  v0.1.28 em-dash panic)
