# v1.2.9

v1.2.9 is a correctness patch on top of v1.2.8's auto-compaction work. It
fixes a premature-compaction bug and a fatal-failure path that together caused
subagents (especially against `claude-opus-4-7`) to die on the first compaction
attempt when the underlying summary call was rate-limited.

## Auto-Compaction

- **Trigger metric is current-prompt size, not cumulative tokens.** Both the
  main agent and subagents now pass `estimate_messages_tokens(&messages)` to
  the compaction predicate (via the new `pub(crate) trigger_tokens` helper),
  matching Claude Code's `tokenCountWithEstimation(messages)`. The previous
  `local_input_tokens.max(estimate_messages_tokens(...))` / `total_input_tokens.max(...)`
  call sites would trip `MICRO_COMPACT_FRACTION (0.65)` after a handful of
  turns even when the real conversation occupied a tiny fraction of the
  effective window.
- **Proactive compaction failure is now soft.** When the structured summary
  call fails (provider error, rate limit, cancellation, invalid summary, or
  no safe boundary), the main agent and subagent runtime now:
  - record the failure on `AutoCompactState` (`on_real_failure`/`on_cancel`),
  - emit a `tracing::warn!` with the actor (`main` / subagent id) and the
    `consecutive_failures` counter,
  - **fall through to the normal provider call within the same turn** with
    unchanged messages.
  This matches Claude Code's "auto-compact failures are retried on the next
  turn" behaviour. After three consecutive failures, the circuit breaker
  (`MAX_COMPACT_FAILURES = 3`) flips `AutoCompactState.disabled = true` and
  proactive compaction silently skips for the rest of the session.
- **Reactive compaction stays fatal.** Compaction triggered by a real
  `context_window_exceeded` error from the provider still surfaces the error
  on failure — the conversation cannot proceed with a still-too-large prompt.
- **Billing counters are decoupled from the trigger.** The cumulative
  `total_input_tokens` (main agent) and the equivalent subagent counter
  remain in place for telemetry but no longer participate in the compaction
  decision, and they are no longer reset to the post-compaction estimate
  after a successful compaction. They monotonically grow over a session.

## Tests

- New regression tests pin the corrected behaviour:
  - `evaluate_compaction_uses_current_size_not_cumulative` (pure-function
    predicate test)
  - `trigger_token_value_is_current_messages_estimate` (call-site test via
    the `pub(crate) trigger_tokens` helper)
  - `proactive_compaction_provider_rate_limit_is_soft_failure` (rate-limited
    summary → `Ok(())` + `consecutive_failures` increments)
  - `reactive_compaction_provider_error_remains_fatal` (force=true → `Err(_)`)
  - `manual_compact_reports_summary_failure_without_synthetic_fallback`
    (manual `/compact` still surfaces provider errors verbatim)
  - `proactive_compaction_failure_falls_through_same_turn` (asserts the
    failing turn still issued the normal subagent provider call)
  - `compaction_summary_request_uses_same_spoof_envelope_as_subagent`
    (Claude Code spoof-mode envelope is byte-identical between
    `request_origin="compaction_summary"` and `request_origin="subagent"`
    when first-user-message is held constant; structural assertions on
    `x-anthropic-billing-header`, `cc_version=`, `cc_entrypoint=`,
    `"You are Claude Code,"`, `x-app: cli`, and `anthropic-beta`)

## Compatibility

- No API surface changes.
- No serialization changes.
- No config changes.
- No data migration.
- Pure behaviour fix; rollback drops the patch and reverts to v1.2.8
  behaviour (premature trigger + fatal proactive failure).
