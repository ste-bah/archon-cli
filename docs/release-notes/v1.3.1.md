# v1.3.1

v1.3.1 adds the first end-to-end JEPA world-model implementation, wires the
world model into ordinary interactive and pipeline guardrails, and adds
backend-native CUDA/MLX accelerator support.

## World Model JEPA

- Added structured trace-window representations, JEPA candidate training,
  masked multi-horizon objectives, EMA target encoder metadata, anti-collapse
  reports, component-loss training ledgers, and JEPA checkpoint persistence.
- Added `train-jepa`, `eval-jepa`, `inspect-jepa`, `compare-representations`,
  and `promote-jepa` world-model commands.
- JEPA promotion uses a fixed local FastEmbed-backed baseline and fails closed
  if that baseline is unavailable or the candidate does not clear promotion
  gates.
- Runtime `model_kind = "jepa_transition"` predictions use JEPA
  context/action encoders plus a persisted transition model; the JEPA predictor
  remains training-only.
- Runtime JEPA failures record typed fail-open reasons such as missing,
  invalid, mismatched, slow, or failed encoder checkpoints.

## Runtime Guardrails

- Added normal-session, coding-task, tool-run, and pipeline-step guardrail
  surfaces so world-model predictions can trigger required verification before
  Archon records a task as complete.
- Guardrails can require tests, builds, lint/typecheck, plan re-checks, critic
  review, or user confirmation depending on learned risk and task policy.
- Interactive runs use turn-completion gating and forced repair turns rather
  than token suppression, so streaming output remains compatible with the TUI.
- Pipeline verification now only treats real execution signals as passing; LLM
  quality scores can no longer satisfy a required test/build action.

## Accelerator Support

- Added CUDA and MLX/Metal JEPA tensor-backend paths for training and runtime
  prediction, with persistent warm device contexts and steady-state latency
  checks.
- Accelerator promotion requires observed per-stage native execution evidence,
  real device identity, tensor self-tests, and zero host fallbacks; backend
  selection alone is not enough to label a candidate CUDA or Metal.
- Added CUDA and MLX build instructions to the JEPA cookbook, including CUDA
  toolkit discovery and macOS Apple Silicon MLX feature builds.

## Safety And Docs

- Existing `latent_transition` behavior remains the default.
- JEPA encoder training uses structured/scalar/categorical features and
  deterministic lexical hashing, not third-party embedding calls.
- Prediction records now include representation metadata for JEPA runs.
- Added a step-by-step world-model and JEPA training cookbook for fresh setups,
  readiness checks, candidate eval, promotion, expected training time, and GPU
  build variants.
- Split oversized world-model/JEPA Rust sources into smaller modules so the
  release code stays within the project code-size rule.
