# v0.1.42 — kill the 5-min subagent timeout footgun

The 300-second wall-clock cap on subagent execution was a footgun for survey/analysis agents (architecture-mapper, system-mapper, codebase-implementation-analyzer, etc.) that legitimately need 10-30+ minutes to scan large codebases and write structured findings to memory. v0.1.42 raises the default to effectively-unlimited, raises the per-spawn `max_turns` ceiling to match, and fixes the misleading error message that caused both LLMs and humans to misdiagnose timeout failures as turn-cap failures.

## What changed

`crates/archon-tools/src/agent_tool.rs::SubagentRequest`:

| Constant | Before | After |
|---|---|---|
| `DEFAULT_TIMEOUT_SECS` | `300` (5 min) | `86_400` (24 h) |
| `MAX_TURNS_HARD_CAP` | `100` (validation upper bound) | `100_000` |
| `DEFAULT_MAX_TURNS` | `10` | `10` (unchanged — quick chat-style subagents stay quick) |

`crates/archon-core/src/subagent.rs::run_loop`:

The wall-clock deadline check now reports both elapsed seconds AND the turn counter, with the actual cap, plus a hint about how to override:

```
Before: "Subagent timed out after 30 turns"
After:  "Subagent wall-clock timeout: 312s elapsed (cap: 300s) at turn 30/40 —
         override per-spawn with timeout_secs:<seconds>, or per-agent in frontmatter"
```

The pre-v0.1.42 message was misleading — the `30` was just the loop counter at the moment of the wall-clock check, not a turn cap. Both LLMs (the in-TUI agent diagnosing its own failure) and human reviewers misread it. The new message surfaces both numbers and identifies which one actually fired.

## Why we removed the cap (rather than tuning it)

The runaway-loop concern the wall-clock cap nominally guarded against is **already covered by `--max-budget-usd`** — that kills on actual cost, not arbitrary time. A wall-clock cap on top of a budget cap is double-jeopardy with no extra safety: it punishes correct behavior (a survey agent that's slow because it's reading a lot, not stuck) without preventing the actual failure mode (a stuck-in-a-loop agent — those still get killed by budget once they burn enough tokens).

If you specifically want a wall-clock cap on a particular agent (e.g. operational reasons — kill anything that's been running > 1h), pass `timeout_secs: 3600` per-spawn, or set it in the agent's frontmatter. The 24-hour default just means there's no opinionated cap for the common case.

## Migration notes

- **No action required for existing agent definitions** — agents that don't specify `timeout_secs` now get 24 h instead of 5 min. Either is correct for any agent that previously ran successfully under the old cap.
- **Per-spawn `max_turns` validation** now allows up to 100,000 (was 100). Heavy survey agents that need >100 turns can request them. The default `max_turns` stays at 10 — only callers that explicitly request more get more.
- **Error message format change** — anything parsing the literal "Subagent timed out after N turns" string will need updating. Internal consumers (the agent activity rail, the TUI failure rendering) read the structured outcome, not the message text, so they're unaffected.

## Verified

```
cargo test -p archon-tools --lib agent_tool::tests::    26 PASS
cargo test -p archon-tools --test task_tools_tests      24 PASS
```

The cap is now effectively unhittable by realistic agent runs. If you DO hit a 24-hour subagent timeout, the agent is genuinely stuck — at which point the meaningful runaway protection (`--max-budget-usd`) will already have fired, since 24 hours of turns burns far more than any reasonable budget cap.
