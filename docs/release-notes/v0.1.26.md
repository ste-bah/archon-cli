# v0.1.26 — GNN auto-retraining (PR #17)

PR 3 of the GNN port saga. Background tokio task wrapping `GnnTrainer`.

## What shipped

- **AutoTrainer** background tokio task with trigger system: new memories (50), corrections (5), elapsed time (6h)
- **Throttle:** 1h minimum between runs, 5min max per run, 256 triplets max per batch
- **First-run kickoff** at startup if existing memory_count > 100
- **NaN/loss-degradation rollback:** training run rolls back to prior weight version if any layer goes NaN/Inf, or if final loss > initial loss × 1.1
- **CozoDB-backed WeightStore** with versioned weight snapshots (`gnn_weights` relation), Adam state persistence (`gnn_adam_state`), training run telemetry (`gnn_training_runs`)
- **`/learning-status`** integration: GNN section showing weight version, last run reason + loss delta, next eligible time, total runs, total rollbacks
- **Hooks** wired into LearningIntegration: `on_agent_complete` increments memory counter, correction-recording site increments correction counter

## Tests

7 new integration tests:
- `gnn_auto_trainer_lifecycle` (2 tests)
- `gnn_auto_trainer_triggers` (3 tests: correction, first_run, memory_accumulation)
- `gnn_auto_trainer_rollback` (1 test)
- `gnn_auto_trainer_does_not_block_foreground` (1 test, the acceptance gate)

## Known issue (fixed in v0.1.27)

The foreground-blocking test was probabilistic — it could pass without actually verifying training overlapped with the measurement window. See [v0.1.27 Finding A](v0.1.27.md).
