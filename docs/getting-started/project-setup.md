# Project setup

archon-cli works against an existing project directory. The `archon-init.sh`
script bootstraps a project layout in any directory: the `.archon/`
config tree, empty `prds/` and `tasks/` directories, and (when invoked
with `--archon-cli-repo` pointing at a clone of archon-cli) a curated
skill + template set.

The script is **POSIX `sh`** (not bash), so it runs on macOS / Linux /
WSL2 / BusyBox without a bash dependency. **Idempotent** — re-running
is safe; existing files are preserved.

## When to run it

- Brand new project: run once at the start.
- Existing project that's never used archon-cli: run once.
- Re-running is idempotent — safe to run again to add missing dirs.
- After a major archon-cli upgrade: re-run with `--archon-cli-repo` to
  pick up newer assets in `assets/skills/` and `assets/templates/`.

## Two scenarios

### Scenario 1: you have only the release binary

You don't have a clone of the archon-cli repo. The installer creates
the directory tree but skips asset copying.

```bash
cd ~/projects/my-project
curl -L https://raw.githubusercontent.com/ste-bah/archon-cli/main/scripts/archon-init.sh -o archon-init.sh
chmod +x archon-init.sh
./archon-init.sh
```

After this, run `archon` and use `/setup-archon-skills` to populate
`~/.config/archon/config.toml` interactively.

### Scenario 2: you cloned the archon-cli repo

You can copy the curated `assets/skills/` and `assets/templates/` into
your project. This is recommended — gives you the full PRD/spec
templates and per-project skill overrides without re-downloading.

```bash
git clone https://github.com/ste-bah/archon-cli ~/code/archon-cli
cd ~/projects/my-project
sh ~/code/archon-cli/scripts/archon-init.sh \
  --target $(pwd) \
  --archon-cli-repo ~/code/archon-cli
```

## Flags

| Flag | Default | Purpose |
|---|---|---|
| `--target <PATH>` | `$PWD` | Project root to initialise |
| `--archon-cli-repo <PATH>` | unset | Path to a clone of archon-cli; enables asset copying from `assets/skills/` and `assets/templates/` |
| `--no-agents` | off | Skip creating `.archon/agents/` directory |
| `-h`, `--help` | — | Print usage and exit |

## What it creates

```
<target>/
├── .archon/
│   ├── agents/        (unless --no-agents)
│   ├── skills/        (gets contents of <repo>/assets/skills/* if --archon-cli-repo set)
│   ├── templates/     (gets contents of <repo>/assets/templates/* if --archon-cli-repo set)
│   ├── adr/           (architecture decision records — written by /grill-with-docs)
│   └── context/       (CONTEXT.md output — written by /grill-with-docs)
├── prds/              (empty; populated by /to-prd)
├── tasks/             (empty; populated by /prd-to-spec)
└── .gitignore         (appends `.archon/` if not present, or creates a new one)
```

## What it does NOT create

The current installer does NOT seed these — they are populated later
by other steps:

| Missing artefact | Created by |
|---|---|
| `.archon/config.toml` | `/setup-archon-skills` wizard, OR copy from `~/.config/archon/config.toml` manually |
| `ARCHON.md` (project instructions file) | Written manually OR generated by a future `/setup-project` skill |
| `.archon/agents/` populated content | Manual copy from a clone, OR future `--archon-cli-repo` upgrade once agent copying lands |

The `--archon-cli-repo` flag currently copies `assets/skills/` and
`assets/templates/` content into `.archon/skills/` and
`.archon/templates/` respectively. **It does NOT copy `.archon/agents/`
content from the source repo** — that gap is tracked for a follow-up.
For now, agents must be copied manually:

```bash
cp -r ~/code/archon-cli/.archon/agents/. ~/projects/my-project/.archon/agents/
```

## Exit codes

| Code | Meaning |
|---|---|
| `0` | success |
| `1` | usage error (unknown flag, missing arg) |
| `2` | target directory does not exist OR is not writable |

## Verifying the layout

```bash
ls -la .archon/ prds/ tasks/
cat .gitignore
```

Expected layout matches the tree in [What it creates](#what-it-creates).

## After running

```bash
archon                    # launch the TUI
```

Then in the TUI:
```
/setup-archon-skills      # 8-step interactive wizard
```

The wizard walks you through API key, model, effort, theme,
permissions, memory provider, and PRD/tasks output paths, writing
answers to `~/.config/archon/config.toml` (user layer). For
project-specific overrides, edit `<workdir>/.archon/config.toml` after
the wizard completes (the installer does NOT create this file —
[Setup wizard](../operations/setup-wizard.md) docs have the schema).

## Troubleshooting

| Symptom | Cause | Fix |
|---|---|---|
| `archon` command not found after install | `~/.cargo/bin` or `~/.local/bin` not in PATH | Add to PATH in shell rc |
| `/setup-archon-skills` doesn't appear in `/skills` | Skill registry stale — happens if you added skills mid-session | Restart `archon`. `/refresh` does NOT reload the skill registry currently — see [Setup wizard](../operations/setup-wizard.md) |
| `.archon/agents/` is empty after `--archon-cli-repo` | Installer doesn't copy agents (known gap) | Manual `cp -r <repo>/.archon/agents/. .archon/agents/` |
| Re-running archon-init.sh wiped my `.gitignore` | It shouldn't — check `git diff` to confirm | The script appends `.archon/` to existing `.gitignore` only if not already present. If clobbered, file an issue |
| Permission denied on `--target` path | Target directory not writable by current user | `chmod +w <target>` or run as the right user |
| `archon-init.sh: target is not a directory` | `--target` points at a file or non-existent path | Pass an existing directory; the script does not create the target itself |
